# SDL Governance — Implementation Plan

Central tools repo for IEC 62443-4-1 aligned Secure Software Development Lifecycle automation. Single source of truth for skills, templates, and CI workflows used across all org project repos.

## Goals

- Minimize developer engagement; automate via agents wherever possible.
- Single source of truth: skills and workflows live here, pulled (not copied) into project repos.
- Project repos stay about the project. Process plumbing is invisible at the repo level.
- Quick human review and sign-off at PR time is the only required human step.

## Per-project repo footprint

The full visible process surface in any project repo:

```
project-repo/
├── docs/sdl/
│   ├── INDEX.md                       # regenerated by sdl-review pre-PR
│   └── YYYY-MM-DD-<branch-slug>/
│       ├── .sdl-meta.yml
│       ├── 01-requirements.md
│       ├── 02-threat-model.md
│       ├── 03-implementation.md
│       └── 04-verification.md
└── .github/workflows/sdl.yml          # 8 lines, calls reusable workflow
```

No CLAUDE.md changes, no AGENTS.md, no copilot-instructions.md, no hooks, no scripts. The presence of `docs/sdl/` is the trigger signal skills look for. CI is the SDL enforcement gate.

`.github/workflows/sdl.yml`:

```yaml
name: sdl
on: [pull_request, push]
jobs:
  validate:
    uses: savioke/sdl/.github/workflows/sdl-validate.yml@v1
```

## Artifact model

Each branch/PR gets `docs/sdl/YYYY-MM-DD-<branch-slug>/` containing:

- `.sdl-meta.yml` — branch, created date, PR number (filled when PR opens), prior-cycle refs.
- `01-requirements.md` — what's being built, security requirements, assets touched, trust boundaries, related prior cycles.
- `02-threat-model.md` — STRIDE-lite, threat IDs, mitigations.
- `03-implementation.md` — what landed, mitigation → commit/file refs.
- `04-verification.md` — checks performed, findings, residual risks.

Templates carry inline IEC 62443-4-1 practice tags as HTML comments (e.g., `## Threats <!-- SR-2 -->`) so each cycle is self-explanatory without dragging the full mapping into every repo.

Branch slug normalization: strip prefix before last `/`, lowercase, underscores → hyphens, truncate to ~40 chars. Original branch name preserved in `.sdl-meta.yml`.

New branch = new SDL cycle. Never delete cycle folders. Cross-reference prior cycles via `01-requirements.md` "Related" field; carry unresolved residual risks forward via `sdl-spec` surfacing them at cycle start.

## Skills

Triggered contextually by the agent based on `SKILL.md` description matching user intent. All three check for `docs/sdl/` in the repo root before acting; absent = no behavior.

**`sdl-spec`** — fires on planning intent ("let's add X", "spec out Y", new feature branch). Detects branch, normalizes slug, scaffolds `docs/sdl/YYYY-MM-DD-<slug>/` from templates, runs a short interview (assets, trust boundaries, data classification, external inputs), populates `01-requirements.md` and `.sdl-meta.yml`. Surfaces unresolved residual risks from prior cycles.

**`sdl-threat-model`** — fires when architecture/design surfaces mid-feature, or after `sdl-spec`. STRIDE-lite per component identified in `01-requirements.md`, populates `02-threat-model.md`. Cross-refs prior cycles' threats to avoid relitigation.

**`sdl-review`** — fires before commit, before PR push, or on "ready for review." Reads diff against merge base. Loads `security-checks.md` category list (new SQL, new endpoints, new deserialization, new crypto, new file I/O with user paths, new external calls, new logging of user data, new auth surfaces, etc.), identifies which apply, then uses model knowledge to verify each per the diff's language. Updates `03-implementation.md` linking mitigations to threat IDs, writes `04-verification.md` with file:line refs and explicit residual-risk entries when uncertain. Regenerates `docs/sdl/INDEX.md`.

No static rule library. Rationale: language-agnostic, scales as models improve, keeps skill prompts small. Auditors see the evidence of review in `04-verification.md`, which is what they actually want.

## Local hooks

None. Considered and rejected: pre-commit warnings and `git log` decoration shims add per-dev setup ceremony for benefits CI already provides. The "dev commits without ever talking to an agent" case is real but cheap — CI catches the missing artifacts at PR time, dev runs the agent, fixes, pushes again. If real usage shows the gap matters, hooks can be added back as opt-in shims via `sync-to-repo.sh`.

## CI validation

`.github/workflows/sdl-validate.yml` calls `lib/validate.py` against the diff:

1. Substantive code changes → matching `.sdl-meta.yml` exists for current branch and was modified in this branch's history.
2. All four artifact files present and not byte-equal to templates.
3. New external inputs / endpoints / SQL / deserialization in diff have corresponding threat IDs in `02-threat-model.md`.
4. `INDEX.md` freshness: warn-only, never fail.

Start strict on (1) and (2); add (3) once skills are stable enough that violations are rare.

## Distribution

Repo is private. Everyone has `git:` access via SSH.

`scripts/install.sh`:
1. `git clone git@github.com:savioke/sdl.git ~/.sdl-governance`
2. `ln -sf ~/.sdl-governance/skills ~/.claude/skills/sdl`
3. `ln -sf ~/.sdl-governance/skills ~/.copilot/skills/sdl`
4. `claude /plugin marketplace add savioke/sdl` for nicer Claude update UX.

Updates: `cd ~/.sdl-governance && git pull`. Symlinks mean every tool sees the new version immediately.

`scripts/sync-to-repo.sh <repo>` (run once per project repo):
1. Copies `.github/workflows/sdl.yml`.
2. Creates `docs/sdl/.gitkeep`.

Both files get committed. New devs cloning a repo that already opted in need no per-clone setup.

## 62443 mapping

`docs/62443-mapping.md` lives only in this repo — it's about the standard, not any project. Templates carry inline practice tags so each cycle is self-documenting at the field level. Audit-time unified report is generated on demand by a small script that walks all repos.

## Build order

1. Templates with inline 62443 practice tags.
2. `docs/62443-mapping.md` and `docs/developer-guide.md`.
3. `sdl-review` skill (`SKILL.md` + `security-checks.md`). Test on past PRs from a Python and a Go repo.
4. `sdl-spec` skill.
5. `sdl-threat-model` skill.
6. `scripts/install.sh`.
7. `scripts/sync-to-repo.sh`.
8. `.github/workflows/sdl-validate.yml` + `lib/validate.py` (structural checks first, semantic later).
9. `marketplace.json` and tag `v1` once exercised on a real feature.

## Honest caveats

- Agent-generated threat models will miss novel domain-specific threats. Treat as a first draft the dev must read and edit. Schedule a quarterly sample audit of recent SDL folders for actual quality.
- 62443-4-1 expects evidence the process is followed and improves over time (SM-2, SM-3). Meaningful commit messages on this repo are themselves audit evidence — treat changes here with the same care as changes to the standards being followed.
